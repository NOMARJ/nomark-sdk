# VUE_STYLE — Vue surface fixture conventions

**Status:** DRAFT — pending owner approval (W3 gate-2)
**Scope:** rules the Vue surface backend (`packages/pro/src/resolvers/vue/backend.ts`) must reproduce byte-for-byte when emitting `expected/vue/*.vue`.
**Authority:** byte-exact equality against `expected/vue/Dashboard.vue`.
**Sibling contract:** REACT_STYLE.md (frozen 2026-04-18 SUR-000). Where Vue's framework idioms diverge from React's, the divergence is documented below; everywhere else the React conventions carry over.

This document will be frozen as contract on owner ack. The backend conforms to it — it does not evolve independently. Changes require a new owner-approval cycle AND a matching fixture-file update in the same commit.

## File container

- Single-file component (`.vue`) with one `<script setup lang="ts">` block and **no** `<template>` block.
- Render is established via `defineRender(() => …)` (Vue 3.5+ macro). The render function returns a single root `VNode` produced by composing the per-verb helpers.
- Rationale: helper-per-verb naming (REACT_STYLE §"Component naming") is the SUR-000 pattern. `<template>` syntax cannot express named-function-per-verb composition; render functions can.

## Strings

- JavaScript string literals: single-quoted — `'fund flows'`. Same as React.
- VNode prop string values: single-quoted — `{ 'data-subject': 'fund flows' }`. Quoted-key form mandatory for any prop containing `-` (`data-*`, `aria-*`).
- Template literals: reserved for interpolation only; never used as a plain-string alternative.

## Indentation

- Two-space indent throughout. No tabs. No trailing whitespace on otherwise-empty lines.
- `<script setup lang="ts">` content starts unindented (column 0), matching SFC convention.

## Imports

- Single `import` from `'vue'` at top of `<script setup>` block.
- `type` specifier used inline for type-only names: `import { h, defineRender, type VNode } from 'vue';`.
- Semi-colon terminates the import statement.

## Types

- `type` over `interface` per repo CLAUDE.md.
- String literal unions — no `enum`.
- Top-level state type is exported from the script block: `export type DashboardState = { … }`.

## Component naming

- Filename — `PascalCase.vue` (e.g., `Dashboard.vue`). The composition name already prefixes the directory; the noun stands alone.
- Per-verb helper functions — `snake_case` matching each `verb.id` from the composition. Not exported. Identical to React.
- No `defineComponent({ name: '…' })` wrapper — `<script setup>` infers the component name from the filename.

## Render-function output

- Each per-verb helper returns `VNode` (or `VNode | null` for conditional verbs — STATUS, GUIDE).
- The top-level `defineRender` callback returns `outcome([…children…].filter((c): c is VNode => c !== null))` — null filtering happens at the composition root, not inside individual helpers.
- Children passed to `h()` are always wrapped in an explicit array, even single children — keeps the emit shape uniform.

## VNode props (round-trip contract)

Same data attribute conventions as React, expressed as object keys instead of JSX attributes:

- `h('main', { 'data-monitor': 'outcome', 'data-subject': 'fund flows', 'data-refresh': '5m' }, children)`
- `h('div', { 'data-arrange': 'grid', 'data-density': 'compact', 'data-children': 'metric_aum,chart_flows' }, children)`
- `h('section', { 'data-display': '<verb.id>', 'data-type': '<params.type>', 'data-emphasis': '<params.emphasis>' }, …)`
- `h('div', { role: 'status', 'aria-live': 'polite', 'data-status': '<params.type>' }, …)`
- `h('div', { role: 'alert', 'data-guide': '<verb.id>', 'data-priority': '<params.priority>' }, …)`

`role` and `aria-live` are unquoted (legal identifiers in JS object literals); every `data-*` and `aria-*` key with a hyphen is quoted.

## Conditional rendering

- Per-verb function returns `null` when the state doesn't satisfy the verb's render predicate (matching React).
- The `defineRender` callback filters `null` from the children array via type-guard predicate `(c): c is VNode => c !== null`.
- No `v-if` directives. No `v-show`. The render-function path keeps all conditional logic in plain TypeScript.

## Comments

- Header block above `<script setup>`, three lines, HTML comment syntax:
  ```vue
  <!-- Generated by NOMARK Intents resolver (target: vue) -->
  <!-- Composition: <name> v<version> -->
  <!-- <description> -->
  ```
- No per-verb function docstrings — verb identity is captured by function name + `data-*` attributes.

## Trailing newline

- Files end with a single `\n`. Matches REACT_STYLE convention.
- No trailing whitespace on any line.

## Formatting discipline

- **No prettier. No eslint post-pass. No AST rewrite.** Raw template-literal emission from the backend with hand-controlled whitespace. Same rule as React.
- Backend is the source of truth; fixture is the byte-exact contract.
- If formatting must change, edit both the backend AND the fixture in the same commit. Never let them drift.

## Vue version pin

- Targets `vue@^3.5` — the version `defineRender` is stable in. Earlier 3.x is unsupported by this backend.
- Listed as a peer dependency on `@nomark-ai/pro` so downstream consumers get a clear resolution failure on older Vue.
