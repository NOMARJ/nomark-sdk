# REACT_STYLE — surface fixture conventions

**Status:** DRAFT — pending owner approval (SUR-000, 2026-04-18)
**Scope:** rules the React surface backend (`packages/pro/src/resolvers/react/backend.ts`) must reproduce byte-for-byte when emitting `expected/react/*.tsx`.
**Authority:** byte-exact equality against `expected/react/Dashboard.tsx`.

Once owner-approved, this document is frozen as contract. The backend conforms to it — it does not evolve independently.

## Strings

- JavaScript string literals: single-quoted — `'fund flows'`.
- JSX attribute string values: single-quoted — `data-subject='fund flows'`. Non-standard vs. React community defaults (which prefer double); chosen for internal consistency with the compute backends' S8 rule.
- Template literals: reserved for interpolation only; never used as a plain-string alternative.

## Indentation

- Two-space indent throughout. No tabs. No indent trailing on otherwise-empty lines.

## Imports

- Single `import` statement from `'react'` at top of file — no default import (modern JSX transform assumed).
- `type` specifier used inline for type-only names: `import { type ReactNode } from 'react';`.
- Semi-colon terminates the import statement.

## Types

- `type` over `interface` per repo CLAUDE.md.
- String literal unions — no `enum`.
- Inline type declarations only; no separate `types.ts` split for v1.
- Top-level state type is exported: `export type DashboardState = { ... }`.

## Component naming

- Top-level composition component — `PascalCase` and exported. Name is the noun — `Dashboard`, not `FundFlowDashboard` (the composition name already prefixes the filename).
- Per-verb helper functions — `snake_case` matching each `verb.id` from the composition. Not exported.

## JSX attribute wrap

- **One attribute, or all-short attributes:** keep on the tag line.
  ```tsx
  <section data-display='metric_aum' data-type='metric' data-emphasis='hero'>
  ```
- **Two or more attributes AND wrapping preferred (readability judgment):** one attribute per line, each indented one level past the tag, closing `>` on its own line:
  ```tsx
  <main
    data-monitor='outcome'
    data-subject='fund flows'
    data-refresh='5m'
  >
  ```
- Trailing comma is NOT used for JSX attributes (invalid syntax). Trailing commas ARE used in TS/TSX call-argument lists, array literals, and object literals — present in the fixture's `Dashboard()` body.

## Self-closing tags

- `<Component />` with trailing space before the slash when no children.
- Elements with children always use explicit open/close tag pairs.

## Conditional rendering

- Per-verb function returns `null` when the state doesn't satisfy the verb's render predicate — e.g., STATUS `loading` returns `null` when `state.status !== 'loading'`.
- Top-level `Dashboard` composes verb outputs with `{fn(state)}`; React renders `null` as nothing.
- No `{cond && <Tag />}` idiom in the fixture — always go through a named function, even for one-line conditionals. Keeps per-verb emission uniform.

## Data attributes (round-trip contract)

- Every verb-derived element carries a `data-<verb-class>='<verb.id>'` attribute:
  - `data-monitor='outcome'`
  - `data-arrange='grid'` (value is `params.type`, since the verb class is implied by the id `layout`)
  - `data-display='<verb.id>'`
  - `data-status='<params.type>'`
  - `data-guide='<verb.id>'`
- Simple scalar params surface as `data-<param-name>='<value>'`:
  - `data-subject='fund flows'` from `MONITOR.subject`
  - `data-refresh='5m'` from `MONITOR.refresh = { value: 5, unit: 'm' }` — joined compactly
  - `data-density='compact'` from `ARRANGE.density`
  - `data-type='metric'` / `data-emphasis='hero'` from `DISPLAY.type` / `DISPLAY.emphasis`
  - `data-priority='primary'` from `GUIDE.priority`
- Arrays collapse to comma-joined strings — `data-children='metric_aum,chart_flows'` from `ARRANGE.children = ['metric_aum', 'chart_flows']`.
- Complex nested params (e.g., `MONITOR.alerts`, `DISPLAY.interact`) are NOT surfaced as data attributes in v1. The backend elides them; they remain available for v2 inspection via the composition IR itself.

## Accessibility baseline (v1)

- `<main>` wraps the MONITOR outcome root.
- `<section>` wraps each DISPLAY emission.
- `<header>` wraps the `label` text inside a DISPLAY when `label` is present.
- `<div role='status' aria-live='polite'>` for STATUS `loading` / `empty` types.
- `<div role='alert'>` for GUIDE emissions.
- Full WCAG 2.2 AA conformance deferred to v2 per PRD open question 3. The v1 ship gate is semantic-HTML parity + role/aria-live on the two transient elements (STATUS, GUIDE).

## Comments

- Header block at top of file, three lines:
  ```tsx
  // Generated by NOMARK Intents resolver (target: react)
  // Composition: <name> v<version>
  // <description>
  ```
- No per-verb function docstrings — verb identity is captured by function name + `data-*` attributes.
- Inline comments permitted where an encoding would surprise a reader (currently none in the v1 fixture — re-evaluate per verb added in v2).

## Trailing newline

- Files end with a single `\n`. Matches TypeScript compute fixture convention.
- No trailing whitespace on any line.

## Formatting discipline

- **No prettier. No eslint post-pass. No AST rewrite.** Raw template-literal emission from the backend with hand-controlled whitespace.
- Backend is the source of truth; fixture is the byte-exact contract.
- If formatting must change, edit both the backend AND the fixture in the same commit. Never let them drift.
