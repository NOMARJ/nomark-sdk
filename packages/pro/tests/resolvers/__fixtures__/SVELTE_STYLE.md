# SVELTE_STYLE — Svelte surface fixture conventions

**Status:** APPROVED — frozen 2026-04-26 (W3 gate-2)
**Scope:** rules the Svelte surface backend (`packages/pro/src/resolvers/svelte/backend.ts`) must reproduce byte-for-byte when emitting `expected/svelte/*.svelte`.
**Authority:** byte-exact equality against `expected/svelte/Dashboard.svelte`.
**Sibling contract:** REACT_STYLE.md (frozen 2026-04-18 SUR-000). Where Svelte's framework idioms diverge from React's, the divergence is documented below; everywhere else the React conventions carry over.

This document will be frozen as contract on owner ack. The backend conforms to it — it does not evolve independently. Changes require a new owner-approval cycle AND a matching fixture-file update in the same commit.

## File container

- Single-file component (`.svelte`). Exactly two `<script>` blocks:
  1. `<script lang="ts" module>` — exports the `DashboardState` type.
  2. `<script lang="ts">` — declares props via the `$props()` rune.
- Below the scripts, the file is a sequence of `{#snippet name(state)}` blocks (one per verb) followed by a single `{@render outcome(state)}` invocation as the render entrypoint.
- Rationale: Svelte 5 snippets (`{#snippet}` + `{@render}`) are the language's named-template-fragment construct. They preserve helper-per-verb naming (REACT_STYLE §"Component naming") natively, without dropping into a render-function dialect.

## Strings

- JavaScript string literals: single-quoted. Same as React/Vue.
- Element attribute values inside snippets: single-quoted — `data-subject='fund flows'`. Same as React's JSX rule.

## Indentation

- Two-space indent throughout. No tabs. No trailing whitespace on otherwise-empty lines.
- `<script>` block contents indented one level past the `<script>` tag (matching SFC convention from Svelte's official examples).
- Snippet bodies indented one level past the opening `{#snippet …}`.

## Imports & types

- `<script lang="ts" module>` block exports types only — no side-effecting imports. The state type lives here so external code (`import type { DashboardState } from './Dashboard.svelte'`) can consume it.
- `<script lang="ts">` block uses Svelte 5 runes: `let { state }: { state: DashboardState } = $props();`.
- No `interface`. No `enum`. Same as React/Vue.

## Component naming

- Filename — `PascalCase.svelte` (e.g., `Dashboard.svelte`).
- Per-verb snippets — `snake_case` matching each `verb.id` from the composition. Identical to React's helper-per-verb convention.
- The composition root is invoked via `{@render outcome(state)}` at the bottom of the file — explicit, single render entry point.

## Snippet output

- Each per-verb snippet receives `(state: DashboardState)` as its only parameter, even snippets that don't read state — keeps the call shape uniform.
- Conditional verbs (STATUS, GUIDE) wrap their entire body in `{#if …}` and emit nothing in the false branch (Svelte compiles to absent DOM, semantically equivalent to React's `null` return).
- Composition verbs (MONITOR, ARRANGE) call `{@render <child>(state)}` for each child snippet inside their root element.

## Element attribute conventions

Same data attribute round-trip contract as React/Vue:

- `<main data-monitor='outcome' data-subject='fund flows' data-refresh='5m'>`
- `<div data-arrange='grid' data-density='compact' data-children='metric_aum,chart_flows'>`
- `<section data-display='<verb.id>' data-type='<params.type>' data-emphasis='<params.emphasis>'>`
- `<div role='status' aria-live='polite' data-status='<params.type>'>`
- `<div role='alert' data-guide='<verb.id>' data-priority='<params.priority>'>`

Multi-attribute wrap follows REACT_STYLE §"JSX attribute wrap":
- One attribute or all-short attributes — keep on the tag line.
- Two or more attributes preferring readability — one per line, each indented one level past the tag, closing `>` on its own line.

## Conditional rendering

- `{#if cond}…{/if}` for STATUS / GUIDE predicates (Svelte's idiomatic conditional). No `{#if cond}…{:else}…{/if}` — the snippet is empty in the false branch.
- No `class:` directives, no `style:` directives — same minimalism as React's "no className, no style props".

## Comments

- Header block at the very top of the file, three lines, HTML comment syntax (Svelte parses these as plain HTML comments before the script blocks):
  ```svelte
  <!-- Generated by NOMARK Intents resolver (target: svelte) -->
  <!-- Composition: <name> v<version> -->
  <!-- <description> -->
  ```
- No `<!-- @component … -->` doc blocks. Verb identity is captured by snippet name + `data-*` attributes.

## Trailing newline

- Files end with a single `\n`.
- No trailing whitespace on any line.

## Formatting discipline

- **No prettier. No svelte-formatter post-pass. No AST rewrite.** Raw template-literal emission from the backend with hand-controlled whitespace.
- Backend is the source of truth; fixture is the byte-exact contract.

## Svelte version pin

- Targets `svelte@^5.0` — `{#snippet}` and `$props()` rune are Svelte 5 features.
- Listed as a peer dependency on `@nomark-ai/pro` so downstream consumers get a clear resolution failure on Svelte 4.
